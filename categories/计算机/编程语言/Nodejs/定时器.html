timer模块

timer 模块开放了一个全局的 API，用于安排函数在未来某个时间点被调用。 因为定时器函数是全局的，所以使用 API 不需要调用 require('timers')。

定时器不阻止退出

调用unref()后，定时器函数将不再阻止程序退出。如果调用unref()后Node发现没有其他任务正在等待或执行，就会直接退出。定时器函数仍在运行，只是它的状态不会再阻止程序退出。
clearTimeout和clearInterval能够实现类似的结果。

setImmediate

callback是一个函数，Node.js 事件循环的此回合结束时进行调用。
...args是调用 callback 时传入的可选的参数。

当多次调用 setImmediate() 时， callback 函数会按它们被创建的顺序放入排队等待执行。 每轮的事件循环迭代都会处理整个回调队列。 如果一个 immediate 定时器是从一个正在执行中的回调内部被放入队列，则该定时器将不会被触发，直到下一轮的事件循环迭代。

事件循环

不同定时器设置的回调函数执行的阶段：
https://nodejs.org/zh-cn/docs/guides/event-loop-timers-and-nexttick/
https://nodejs.org/en/docs/guides/event-loop-timers-and-nexttick/

每个阶段都有一个执行队列。idle, prepare阶段仅供内部使用，用户不会看到这个阶段。
一般（poll阶段例外）当执行队列为空，或者执行的回调达到了最大上限，就进入下一个阶段。

  1. timers阶段执行 setTimeout() 和 setInterval() 的回调函数，前提是已经到达这两个定时器函数设置触发时间。若没有回调函数，或者没有到达时间的回调函数，则进入下一阶段。
  2. pending callbacks阶段执行某些系统操作的回调，例如TCP错误。
  3. poll阶段处理大部分回调函数（只要其他阶段不负责执行，例如I/O），当处理完后：
    1. 如果timers队列不存在到达时间的回调，且check队列（setImmediate设置的回调）为空，那么就一直维持在poll阶段等待执行新的回调函数。
    2. 否则，进入下一个阶段。
  4. check阶段执行setImmediate设置的回调。
  5. close callbacks阶段：如果套接字或句柄突然关闭，例如socket.destroy()，则在此阶段执行close事件的回调。

process.nextTick()不是事件循环的一部分，不管事件循环处于哪个阶段，nextTick队列都会在当前任务（回调函数）完成后处理，如果当前阶段的队列有多个回调函数，nextTick会在一个回调函数结束后就立刻执行，从而推迟其他回调函数的执行。

setImmediate()和setTimeout
哪个定时器执行取决于调用上下文，如果都是在主模块调用的，则时机受到进程性能影响（影响因素包括机器上的其他程序）。
使用 setImmediate() 相对于setTimeout() 的主要优势是，如果setImmediate()是在 I/O 周期内（poll轮询）被调度的，那它将会在其中任何的定时器之前执行，跟这里存在多少个定时器无关
