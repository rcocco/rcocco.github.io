25.1 迭代

25.1.1 公共迭代接口

接口是一组属性键，其对应的值符合特定的规范。 提供了接口规范描述的全部属性的对象，称为此对象符合该接口。一个接口不是由一个单独的对象来表示的。可能有许多单独实现的对象符合任何接口。一个单独的对象可能符合多个接口。

25.1.1.1 Iterable接口

Iterable接口包含表65中描述的属性：

表65：Iterable接口必需的属性

属性 值 要求
@@iterator 返回Iterator对象的函数。 返回的对象必须符合Iterator接口。

25.1.1.2 Iterator接口

实现Iterator接口的对象必须包括表66中的属性。此类对象还可以实现表67中的属性。

表66：Iterator接口必需的属性

属性 值 要求
next 返回IteratorResult对象的函数。 返回的对象必须符合IteratorResult接口。 如果对Iterator的next方法的前一次调用返回了done属性为true的IteratorResult对象，则对该对象的next方法的所有后续调用也应返回其done属性为true的IteratorResult对象。 但是，此要求不是强制的。

注：可以将参数传递给next函数，但是参数的解释和有效性取决于目标Iterator。 for-of语句和Iterators的其他普通用户不传递任何参数，因此，期望以这种方式使用的Iterator对象必须准备好应对不带参数的调用。

表67：Iterator接口可选的属性

属性 值 要求
return 返回IteratorResult对象的函数。 返回的对象必须符合IteratorResult接口。 调用此方法会通知Iterator对象，调用者无意再对Iterator进行next方法调用。 返回的IteratorResult对象通常具有一个done属性（值为true）和一个value属性（值为传递给return方法的参数）。 但是，此要求不是强制的。
throw 返回IteratorResult对象的函数。 返回的对象必须符合IteratorResult接口。 调用此方法会通知Iterator对象调用者已检测到错误情况。 参数可用于标识错误条件，并且通常是异常对象。 典型的响应是throw作为参数传递的值。 如果该方法没有throw，则返回的IteratorResult对象通常将具有一个值为true的done属性。

注2：通常，这些方法的调用者应在调用它们之前检查它们的存在。 某些ECMAScript语言功能（包括for-of，yield *和数组解构）在执行存在性检查后调用这些方法。 大多数接受Iterable对象作为参数的ECMAScript库函数也会有条件地调用它们。

25.1.1.4 AsyncIterator接口

实现 AsyncIterator 接口的对象必须包括表69中的属性。此类对象还可以实现表70中的属性。

表69：AsyncIterator接口必需的属性

属性 值 要求
next 返回Promise的函数(结果值是IteratorResult对象) 返回的Promise在兑现时，必须用符合IteratorResult接口的对象来兑现。如果对AsyncIterator的next方法的前一次调用返回了一个兑现值为IteratorResult对象的Promise，且这个IteratorResult对象的done属性为true，则对该AsyncIterator的next方法的所有后续调用也应返回一个兑现值为IteratorResult对象的Promise，且这个IteratorResult的done属性为true。 但是，此要求不是强制的。此外，作为兑现值的IteratorResult对象应该有value属性，其值不是Promise(或thenable)，然而，这一要求也不是强制的。

注：可以将参数传递给next函数，但是参数的解释和有效性取决于目标AsyncIterator。 for-await-of语句和Iterators的其他普通用户不传递任何参数，因此，期望以这种方式使用的AsyncIterator 对象必须准备好应对不带参数的调用。

表70：AsyncIterator 接口可选的属性

属性 值 要求
return 返回Promise的函数(结果值是IteratorResult对象) 返回的Promise在兑现时，必须用符合IteratorResult接口的对象来兑现。 调用此方法会通知AsyncIterator对象，调用者无意再对AsyncIterator进行next方法调用。 返回的Promise的兑现值IteratorResult对象通常具有一个done属性（值为true）和一个value属性（值为传递给return方法的参数）。 但是，此要求不是强制的。此外，作为兑现值的IteratorResult对象应该有value属性，其值不是Promise(或thenable)。如果以通常的方式使用参数值，那么当它是一个拒绝的Promise时，应该返回一个用同样结果值拒绝的Promise；当它是一个解决的Promise时，它的兑现值应该被用作返回的Promise的兑现值IteratorResult对象的value属性。然而，这一要求也不是强制的。
throw 返回Promise的函数(结果值是IteratorResult对象) 返回的Promise在兑现时，必须用符合IteratorResult接口的对象来兑现。 调用此方法会通知AsyncIterator对象调用者已检测到错误情况。 参数可用于标识错误条件，并且通常是异常对象。 典型的响应是返回一个拒绝的Promise，其拒绝值是参数传递的值。 如果返回的Promise是已解决的，那么作为兑现值的IteratorResult对象通常具有值为true的done属性。此外，它也应该有一个value属性，其值不是Promise(或thenable)，然而，这一要求也不是强制的。

注2：通常，这些方法的调用者应在调用它们之前检查它们的存在。 某些ECMAScript语言功能（包括for-await-of，yield *和数组解构）在执行存在性检查后调用这些方法。

25.1.1.5 IteratorResult接口

IteratorResult接口包含表71中列出的属性：
表71：IteratorResult接口属性

属性 值 要求
done true或false 这是迭代器next方法调用的结果状态。 如果到达迭代器的终点，则done为true。 如果未达到终点，则done为false，并且可以使用值。 如果done属性（自有的或继承的）不存在，则认为该属性的值为false。
value 任何ECMAScript语言值 如果done为false，则此值为当前迭代元素值。 如果done为true，则此值为迭代器的返回值（如果提供了）。 如果迭代器没有返回值，则value为undefined。 在那种情况下，如果符合属性的对象没有继承显式的value属性，则可能缺少value属性。

25.1.2 %IteratorPrototype%对象

%IteratorPrototype%对象：

  • 具有[[Prototype]]内部槽，且其值为%Object.prototype%
  • 是普通对象

注：本规范中定义的所有实现Iterator接口的对象也都继承自%IteratorPrototype%。ECMAScript代码也可以定义继承自%IteratorPrototype%的对象（但不能通过类定义的extends子句，因为%IteratorPrototype%不是构造函数）。%IteratorPrototype%对象提供了一个可以添加适用于所有迭代器对象的附加方法的地方。

以下表达式是ECMAScript代码访问%IteratorPrototype%对象的一种方式：

25.1.4 Async-from-Sync迭代器对象

Async-from-Sync Iterator对象是一个异步迭代器，它适配一个特定的同步迭代器。对于Async-from-Sync迭代器对象，没有一个命名的构造函数。相反，Async-from-Sync迭代器对象是根据需要由CreateAsyncFromSyncIterator抽象操作创建。

25.1.4.1 CreateAsyncFromSyncIterator ( syncIteratorRecord )

抽象操作CreateAsyncFromSyncIterator用于从同步迭代器记录创建异步迭代器记录，它执行以下步骤：

  1. 令asyncIterator为! OrdinaryObjectCreate(%AsyncFromSyncIteratorPrototype%, « [[SyncIteratorRecord]] »)
  2. 将asyncIterator.[[SyncIteratorRecord]]设为syncIteratorRecord
  3. 令nextMethod为! Get(asyncIterator, "next")
  4. 令iteratorRecord为记录{ [[Iterator]]: asyncIterator, [[NextMethod]]: nextMethod, [[Done]]: false }
  5. 返回iteratorRecord

25.1.4.2 %AsyncFromSyncIteratorPrototype%对象

%AsyncFromSyncIteratorPrototype%对象：

  • 具有被所有Async-from-Sync迭代器对象继承的属性
  • 是普通对象
  • 具有[[Prototype]]内部槽，且其值为%AsyncIteratorPrototype%
  • 具有以下属性
25.1.4.2.1 %AsyncFromSyncIteratorPrototype%.next ( value )
  1. 令O为this值
  2. 断言：Type(O)是Object，且O具有[[SyncIteratorRecord]]内部槽
  3. 令promiseCapability为! NewPromiseCapability(%Promise%)
  4. 令syncIteratorRecord为O.[[SyncIteratorRecord]]
  5. 令result为IteratorNext(syncIteratorRecord, value)
  6. IfAbruptRejectPromise(result, promiseCapability)
  7. 返回! AsyncFromSyncIteratorContinuation(result, promiseCapability)
25.1.4.2.2 %AsyncFromSyncIteratorPrototype%.return ( value )
  1. 令O为this值
  2. 断言：Type(O)是Object，且O具有[[SyncIteratorRecord]]内部槽
  3. 令promiseCapability为! NewPromiseCapability(%Promise%)
  4. 令syncIterator为O.[[SyncIteratorRecord]].[[Iterator]]
  5. 令return为GetMethod(syncIterator, "return")
  6. IfAbruptRejectPromise(result, promiseCapability)
  7. 如果return为undefined，则：
    1. 令iterResult为! CreateIterResultObject(value, true)
    2. 执行! Call(promiseCapability.[[Resolve]], undefined, « iterResult »)
    3. 返回promiseCapability.[[Promise]]
  8. 令result为Call(return, syncIterator, « value »)
  9. IfAbruptRejectPromise(result, promiseCapability)
  10. 如果Type(result)不是Object，则：
    1. 执行! Call(promiseCapability.[[Reject]], undefined, « 新创建的TypeError对象 »)
    2. 返回promiseCapability.[[Promise]]
  11. 返回! AsyncFromSyncIteratorContinuation(result, promiseCapability)
25.1.4.2.3 %AsyncFromSyncIteratorPrototype%.throw ( value )
  1. 令O为this值
  2. 断言：Type(O)是Object，且O具有[[SyncIteratorRecord]]内部槽
  3. 令promiseCapability为! NewPromiseCapability(%Promise%)
  4. 令syncIterator为O.[[SyncIteratorRecord]].[[Iterator]]
  5. 令throw为GetMethod(syncIterator, "throw")
  6. IfAbruptRejectPromise(result, promiseCapability)
  7. 如果throw为undefined，则：

25.4.3.1 GeneratorStart ( generator, generatorBody )

接受参数generator和generatorBody的抽象操作GeneratorStart执行以下步骤：

  1. 断言：generator.[[GeneratorState]]的值为undefined
  2. 令genContext为运行中的执行上下文
  3. 将genContext的Generator组件设为generator
  4. 设置genContext的代码计算状态，使得恢复对该执行上下文的计算时，执行以下步骤：
    1. 令result为计算generatorBody的结果
    2. 断言：如果在这里返回，则generator要么抛出异常，要么执行一个隐式或显式的return
    3. 从执行上下文栈中删除genContext，并且将执行上下文栈的栈顶恢复成运行中的执行上下文
    4. 将generator.[[GeneratorState]]设为completed
    5. 一旦生成器进入completed状态，它就永远不会离开，其相关的执行环境也不会恢复。任何与generator相关的执行状态在此时都可以被丢弃。
    6. 如果result.[[Type]]为normal，令resultValue为undefined
    7. 否则如果result.[[Type]]为return，令resultValue为result.[[Value]]
    8. 否则：
      1. 断言：result.[[Type]]为throw
      2. 返回Completion(result)
    9. 返回CreateIterResultObject(resultValue, true)
  5. 将generator.[[GeneratorContext]]设为genContext
  6. 将generator.[[GeneratorState]]设为suspendedStart
  7. 返回NormalCompletion(undefined)

25.4.3.2 GeneratorValidate ( generator )

抽象操作GeneratorValidate接受参数generator并执行以下步骤：

  1. 执行? RequireInternalSlot(generator, [[GeneratorState]])
  2. 断言：generator也具有[[GeneratorContext]]内部槽
  3. 令state为generator.[[GeneratorState]]
  4. 如果state为executing，抛出 TypeError 异常
  5. 返回state

25.4.3.3 GeneratorResume ( generator, value )

抽象操作GeneratorResume接受参数generator和value并执行以下步骤：

  1. 令state为? GeneratorValidate(generator)
  2. 如果state为completed，返回CreateIterResultObject(undefined, true)
  3. 断言：state要么是suspendedStart，要么是suspendedYield
  4. 令genContext为generator.[[GeneratorContext]]
  5. 令methodContext为运行中的执行上下文
  6. 暂停methodContext
  7. 将generator.[[GeneratorState]]设为executing
  8. 将genContext推送到执行上下文栈。genContext现在是运行中的执行上下文
  9. 恢复暂停的genContext的计算，使用NormalCompletion(value)作为暂停该计算的操作的结果。令result为恢复的计算所返回的完结记录
  10. 断言：当我们返回这个位置时，genContext已经被从执行上下文栈移除了，同时methodContext现在是运行中的执行上下文
  11. 返回Completion(result)

25.4.3.4 GeneratorResumeAbrupt ( generator, abruptCompletion )

抽象操作GeneratorResumeAbrupt接受参数generator和abruptCompletion并执行以下步骤：

  1. 令state为? GeneratorValidate(generator)
  2. 如果state为suspendedStart，则：
    1. 将generator.[[GeneratorState]]设为completed
    2. 一旦生成器进入completed状态，它就永远不会离开，其相关的执行环境也不会恢复。任何与generator相关的执行状态在此时都可以被丢弃。
    3. 将state设为completed
  3. 如果state为completed，则：
    1. 如果abruptCompletion.[[Type]]为return，则返回CreateIterResultObject(abruptCompletion.[[Value]], true)
    2. 返回Completion(abruptCompletion)
  4. 断言：state是suspendedYield
  5. 令genContext为generator.[[GeneratorContext]]
  6. 令methodContext为运行中的执行上下文
  7. 暂停methodContext
  8. 将generator.[[GeneratorState]]设为executing
  9. 将genContext推送到执行上下文栈。genContext现在是运行中的执行上下文
  10. 恢复暂停的genContext的计算，使用abruptCompletion作为暂停该计算的操作的结果。令result为恢复的计算所返回的完结记录。
  11. 断言：当当我们返回这个位置时，genContext已经被从执行上下文栈移除了，同时methodContext现在是运行中的执行上下文
  12. 返回Completion(result)

  1. 令genContext为运行中的执行上下文

  5. 否则，返回sync

  2. 断言：generator.[[GeneratorState]]的值为undefined
  3. 令genContext为运行中的执行上下文
  4. 将genContext的Generator组件设为generator
  5. 设置genContext的代码计算状态，使得恢复对该执行上下文的计算时，执行以下步骤：
    1. 令result为计算generatorBody的结果
    2. 断言：如果返回到这里，则异步生成器要么抛出异常，要么执行一个隐式或显式的return

    6. 否则：
      1. 令resultValue为result.[[Value]]
      2. 如果result.[[Type]]不为return，则返回! AsyncGeneratorReject(generator, resultValue)
    7. 返回! AsyncGeneratorResolve(generator, resultValue, true)
  6. 将generator.[[AsyncGeneratorContext]]设为genContext
  7. 将generator.[[AsyncGeneratorState]]设为suspendedStart
  8. 将generator.[[AsyncGeneratorQueue]]设为一个空列表

  2. 令queue为generator.[[AsyncGeneratorQueue]]

  1. 断言：generator是AsyncGenerator实例
  2. 令state为generator.[[AsyncGeneratorState]]
  3. 断言：state不是executing
  4. 如果state是awaiting-return，返回undefined
  5. 令queue为generator.[[AsyncGeneratorQueue]]
  6. 如果queue是空列表，返回undefined
  7. 令next为queue的一个元素的值
  8. 断言：next是AsyncGeneratorRequest记录
  9. 令completion为next.[[Completion]]
  10. 如果completion是非常规完结，则：
    1. 如果state是suspendedStart，则：
      1. 将generator.[[AsyncGeneratorState]]设为completed
      2. 将state设为completed
    2. 如果state是completed，则：
      1. 如果completion.[[Type]]是return，则：
        1. 将generator.[[AsyncGeneratorState]]设为awaiting-return
        2. 令promise为? PromiseResolve(%Promise%, completion.[[Value]])

      2. 否则：
        1. 断言：completion.[[Type]]为throw
        2. 执行! AsyncGeneratorReject(generator, completion.[[Value]])
        3. 返回undefined
  11. 否则如果state是completed，返回! AsyncGeneratorResolve(generator, undefined, true)
  12. 断言：state要么是suspendedStart，要么是suspendedYield
  13. 令genContext为generator.[[AsyncGeneratorContext]]
  14. 令callerContext为运行中的执行上下文

  19. 断言：result永远不会是非常规完结
  20. 断言：当我们回到这里时，genContext已经从执行上下文栈移除了，并且callerContext现在是运行中的执行上下文
  21. 返回undefined

25.6 Promise对象

Promise是一个对象，它被用作延迟（可能还是异步）计算的最终结果的占位符。

任何Promise对象均处于以下三种互斥状态之一：已兑现(fulfilled)，已拒绝(rejected)和待定(pending)：

  • 如果p.then(f, r)立即使Job入队来调用函数f，则Promisep是已兑现的
  • 如果p.then(f, r)立即使Job入队来调用函数r，则Promisep是已拒绝的
  • 如果Promise既不是已兑现的，也不是已解决的，则它是待定的

如果Promise不是待定的，则被称为确定的(settled)，也就是说它要么已兑现要么已拒绝

如果一个Promise是确定的，或者已被“锁定(locked in)”以匹配另一个Promise的状态，则该Promise是已解决的(resolved)。尝试resolve（解决）或reject（拒绝）一个已解决的Promise没有任何效果。如果Promise不是已解决的，则它是未解决的。一个未解决的Promises总是待定状态。一个已解决的Promise可能是待定的，也可能是已兑现的，还可能是已拒绝的

    1. 令job为NewPromiseReactionJob(reaction, argument)
    2. 执行HostEnqueuePromiseJob(job.[[Job]], job.[[Realm]])
  2. 返回undefined

25.6.1.9 HostPromiseRejectionTracker ( promise, operation )

HostPromiseRejectionTracker是一个实现定义的抽象操作，它允许主机环境跟踪Promise的拒绝。

HostPromiseRejectionTracker的实现必须在所有情况下正常完结。HostPromiseRejectionTracker的默认实现是无条件地返回一个空的正常完结。

注1：HostPromiseRejectionTracker在两种情况下被调用：

  • 当一个promise在没有任何处理程序的情况下被拒绝时，使用"reject"作为参数operation来调用它。
  • 当一个处理程序第一次被添加到一个被拒绝的promise中时，使用"handle"作为参数operation来调用它。
    HostPromiseRejectionTracker的典型实现可能会尝试通知开发者未处理的拒绝，同时也会小心地通知他们，如果这种先前的通知后来因新的处理程序被附加而失效。

注2：如果operation是 "handle"，实现不应该以干扰垃圾回收的方式持有对promise的引用。如果operation是 "reject"，实现可以持有对promise的引用，因为预计拒绝将是罕见的，并且不在热代码路径上。

25.6.2 Promise Jobs

25.6.2.1 NewPromiseReactionJob ( reaction, argument )

抽象操作NewPromiseReactionJob接受两个参数：reaction和argument，返回一个新的Job抽象闭包，该闭包将适当的处理程序应用于incoming值，并使用处理程序的返回值来解决或拒绝与该处理程序关联的派生promise。它执行以下步骤：

  1. 令job为一个新的Job抽象闭包，该闭包不接受参数，捕获reaction和argument，并在调用时执行以下步骤：
    1. 断言：reaction为一个新的PromiseReaction记录
    2. 令promiseCapability为reaction.[[Capability]]
    3. 令type为reaction.[[Type]]
    4. 令handler为reaction.[[Handler]]
    5. 如果handler为undefined，则：
      1. 如果type是Fulfill，令handlerResult为NormalCompletion(argument)
      2. 否则，
        1. 断言：type为Reject
        2. 令handlerResult为ThrowCompletion(argument)
    6. 否则， 令handlerResult为Call(handler, undefined, « argument »)
    7. 如果promiseCapability为undefined，则：
      1. 断言：handlerResult不是非常规完结
      2. 返回NormalCompletion(empty)
    8. 如果handlerResult是非常规完结，则令status为Call(promiseCapability.[[Reject]], undefined, « handlerResult.[[Value]] »)
    9. 否则，令status为Call(promiseCapability.[[Resolve]], undefined, « handlerResult.[[Value]] »)
    10. 返回Completion(status)
  2. 令handlerRealm为null
  3. 如果reaction.[[Handler]]不是undefined，则：
    1. 令getHandlerRealmResult为GetFunctionRealm(reaction[[Handler]]).
    2. 如果getHandlerRealmResult是常规完结，则将handlerRealm设为getHandlerRealmResult.[[Value]]
  4. 返回记录{ [[Job]]: job, [[Realm]]: handlerRealm }

25.6.2.2 NewPromiseResolveThenableJob ( promiseToResolve, thenable, then)

抽象操作NewPromiseResolveThenableJob接受三个参数：promiseToResolve、thenable和then，并执行以下步骤：

  1. 令job为一个新的Job抽象闭包，该闭包不接受参数，捕获promiseToResolve、thenable和then，并在调用时执行以下步骤：
    1. 令resolvingFunctions为CreateResolvingFunctions(promiseToResolve)
    2. 令thenCallResult为Call(then, thenable, «resolvingFunctions.[[Resolve]], resolvingFunctions.[[Reject]]»)
    3. 如果thenCallResult是非常规完结，则
      1. 令status为Call(resolvingFunctions.[[Reject]], undefined, «thenCallResult.[[value]]»)
      2. 返回Completion(status)
    4. 返回Completion(thenCallResult)
  2. 令getThenRealmResult为GetFunctionRealm(then)
  3. 如果getThenRealmResult是常规完结，则令thenRealm为getThenRealmResult.[[Value]]
  4. 否则，令thenRealm为null
  5. 返回记录{ [[Job]]: job, [[Realm]]: thenRealm }

注：此Job使用提供的thenable及其then方法来解决给定的promise。 此过程必须作为Job进行，以确保对所有周围代码的计算完成后，再对then方法进行计算。

25.6.3 Promise构造函数

Promise构造函数：

  • 是固有对象%Promise%
  • 是全局对象的"Promise"属性的初始值
  • 当作为构造函数调用时，它将创建并初始化一个新的Promise对象
  • 不能作为函数调用，以这种方式调用时会抛出异常。
  • 被设计为可子类化的。 它可以用作类定义的extends子句中的值。 打算继承指定Promise行为的子类构造函数必须包括对Promise构造函数的super调用，以便使用支持Promise和Promise.prototype内置方法所必需的内部状态来创建和初始化子类实例。

25.6.3.1 Promise ( executor )

使用参数executor来调用函数Promise时，执行以下步骤：

  1. 如果NewTarget为undefined，抛出 TypeError 异常
  2. 如果IsCallable(executor)为false，抛出 TypeError 异常
  3. 令promise为OrdinaryCreateFromConstructor(NewTarget, "%PromisePrototype%", «‍[[PromiseState]], [[PromiseResult]], [[PromiseFulfillReactions]], [[PromiseRejectReactions]]» )
  4. 将promise.[[PromiseState]]的值设为pending
  5. 将promise.[[PromiseFulfillReactions]]的值设为一个新的空列表
  6. 将promise.[[PromiseRejectReactions]]的值设为一个新的空列表
  7. 将promise.[[PromiseIsHandled]]的值设为false
  8. 令resolvingFunctions为CreateResolvingFunctions(promise)
  9. 令completion为Call(executor, undefined, «resolvingFunctions.[[Resolve]], resolvingFunctions.[[Reject]]»)
  10. 如果completion是非常规完结，则执行? Call(resolvingFunctions.[[Reject]], undefined, «completion.[[value]]»)
  11. 返回promise

注：executor参数必须是一个函数对象。 调用该函数以初始化和报告此Promise对象所表示的延迟操作的完成。 executor接受两个参数：resolve和reject。 这两个函数被executor函数使用，以便报告延迟计算最终完成或者失败。从executor函数返回并不意味着延迟的动作已经完成，它仅表示接受了一个请求：未来会执行该延迟动作。

传递给executor函数的resolve函数接受一个参数。executor函数的代码最终可能会调用resolve函数来指示它希望解决关联的Promise对象。传递给resolve函数的参数表示延迟动作的最终值，可以是实际的兑现值，也可以是另一个Promise对象（当它兑现时则提供该值）

传递给executor函数的reject函数接受一个参数。executor函数的代码最终可能会调用reject函数来指示关联的Promise对象被拒绝了，并且永远不会兑现。传递给reject函数的参数用作promise的拒绝值，通常是一个Error对象。

Promise构造函数传递给executor函数的resolve和reject函数具有实际确定和拒绝关联的promise的能力。 子类可能具有不同的构造函数行为，这些构造函数将传入自定义值以进行确定和拒绝。

  4. 如果IsCallable(onFulfilled)为false，则令onFulfilled为"undefined"
  5. 如果IsCallable(onRejected)为false，则令onRejected为"undefined"
  6. 令fulfillReaction为PromiseReaction { [[Capability]]: resultCapability, [[Type]]: Fulfill, [[Handler]]: onFulfilled }
  7. 令rejectReaction为PromiseReaction { [[Capability]]: resultCapability, [[Type]]: Reject, [[Handler]]: onRejected }
  8. 如果promise.[[PromiseState]]的值为pending，则：
    1. 将fulfillReaction作为最后一个元素添加到列表promise.[[PromiseFulfillReactions]]
    2. 将rejectReaction作为最后一个元素添加到列表promise.[[PromiseRejectReactions]]
  9. 否则如果promise.[[PromiseState]]的值为fulfilled，则
    1. 令value为promise.[[PromiseResult]]
    2. 令fulfillJob为NewPromiseReactionJob(fulfillReaction, value)
    3. 执行HostEnqueuePromiseJob(fulfillJob.[[Job]], fulfillJob.[[Realm]])
  10. 否则：
    1. 断言：promise.[[PromiseState]]的值为rejected
    2. 令reason为promise.[[PromiseResult]]
    3. 如果promise.[[PromiseIsHandled]]为false，执行HostPromiseRejectionTracker(promise, "handle")
    4. 令rejectJob为NewPromiseReactionJob(rejectReaction, reason)
    5. 执行HostEnqueuePromiseJob(rejectJob.[[Job]], rejectJob.[[Realm]])
  11. 如果resultCapability为undefined，则返回undefined
  12. 否则返回resultCapability.[[Promise]]

25.6.5.5 Promise.prototype [ @@toStringTag ]

@@toStringTag属性的初始值为字符串"Promise"

此属性具有特性{ [[Writable]]: false, [[Enumerable]]: false, [[Configurable]]: true }

25.7 异步函数对象

AsyncFunction对象是通常通过计算AsyncFunctionDeclarations、AsyncFunctionExpressions、AsyncMethods和AsyncArrowFunctions创建的函数。它们也可以通过调用固有的%AsyncFunction%来创建。

25.7.1 AsyncFunction构造函数

AsyncFunction构造函数：

  • 是固有对象%AsyncFunction%
  • 是Function的子类
  • 作为函数而不是构造函数调用时创建并初始化新的异步函数对象。因此函数调用AsyncFunction(…)等价于接受相同参数的对象创建表达式new AsyncFunction(…)
  • 被设计为可被子类化。它可以被用作类定义中的extends子句的值。打算继承指定的 AsyncFunction行为的子类构造函数必须包括对 AsyncFunction 构造函数的super调用，以创建并初始化一个具有内置 Async 函数行为所需的内部槽的子类实例。

25.7.1.1 AsyncFunction ( p1, p2, … , pn, body )

最后一个参数指定了一个异步函数的主体（可执行代码）。任何前面的参数都指定了形式参数。

当AsyncFunction函数使用参数p1, p2, ..., pn, body（其中n可能是0，也就是没有p参数，而且body也可能没有提供）调用时，会执行以下步骤：

  1. 令C为激活的函数对象
  2. 令args为通过[[Call]]或[[Construct]]传递给此函数的参数列表
  3. 返回CreateDynamicFunction(C, NewTarget, async, args)

注：见 19.2.1.1 的注

25.7.2 AsyncFunction构造函数的属性

  • 具有name属性，其值为"AsyncFunction"
  • 具有下列属性：

25.7.2.1 AsyncFunction.length

此属性具有特性{ [[Writable]]: false, [[Enumerable]]: false, [[Configurable]]: false }

25.7.3 AsyncFunction原型对象的属性

  • 不是函数对象，没有[[ECMAScriptCode]]内部槽，也没有表27中列出的任何其他内部槽

25.7.5 异步函数抽象操作

25.7.5.1 AsyncFunctionStart ( promiseCapability, asyncFunctionBody )

  1. 令runningContext为运行中的执行上下文
  2. 令asyncContext为runningContext的副本
  3. 注意：复制执行状态对于下面的步骤恢复其执行是必需的。恢复当前正在执行的上下文是没有定义的。
  4. 设置asyncContext的代码计算状态，以便在恢复对该执行上下文的计算时，将执行以下步骤：
    1. 令result为计算asyncFunctionBody的结果.
    2. 断言：如果我们在这里返回，异步函数要么抛出异常，要么执行显式或隐式的返回。所有await都完成了。
    3. 从执行上下文栈中移除asyncContext，并将执行上下文栈的栈顶恢复为运行中的执行上下文
    4. 如果result.[[Type]]为normal，则
      1. 执行! Call(promiseCapability.[[Resolve]], undefined, « undefined »)
    5. 否则如果result.[[Type]]为return，则
      1. 执行! Call(promiseCapability.[[Resolve]], undefined, « result.[[Value]] »)
    6. 否则：
      1. 断言：result.[[Type]]为throw
      2. 执行! Call(promiseCapability.[[Reject]], undefined, « result.[[Value]] »)
    7. 返回。
  5. 将asyncContext推入执行上下文栈，asyncContext现在是运行中的执行上下文
  6. 恢复asyncContext的暂停的计算，令result为恢复的计算所返回的值。
  7. 断言：当我们返回这个位置时，asyncContext已经从执行上下文栈中移除了，runningContext现在是运行中的执行上下文
  8. 断言：结果是一个值为undefined的常规完结。完结值的可能来源是Await，或者，如果异步函数没有等待任何东西，那么上面的步骤4.7。
  9. 返回。
