6 ECMAScript数据类型和值

本规范中的算法操作的值都有一个关联的类型。可能的值类型正是本节中定义的那些。类型被进一步细分为ECMAScript语言类型和规范类型。

在本规范中，记号 "Type(x)"被用作 "x的类型 "的缩写，其中 "类型 "指的是本节中定义的ECMAScript语言和规范类型。当术语 "empty "被用来命名一个值时，它相当于说 “没有任何类型的值”。

6.1 ECMAScript语言类型

ECMAScript语言类型 对应的是程序员使用 ECMAScript 语言直接操作的值。ECMAScript语言类型 包括 Undefined、Null、Boolean、String、Symbol、Number、BigInt、Object。ECMAScript语言值是以ECMAScript语言类型为特征的值。

6.1.1 Undefined类型

Undefined类型 有且只有一个值，称为 undefined 。任何没有被赋值的变量值为 undefined 。

6.1.2 Null类型

Null类型 有且只有一个值，称为 null 。

6.1.3 Boolean类型

Boolean类型 表示逻辑实体，有两个值，称为 true 和 false。

6.1.4 String类型

字符串类型是零个或多个16位无符号整数值（“元素”）的所有有序序列的集合，最大长度为2^53-1个元素。在运行中的 ECMAScript 程序中，字符串类型常被用于表示文本数据，此时字符串中的每个元素都被视为一个 UTF-16 代码单元值。每个元素都被认为占有此序列中的一个位置，用非负整数索引这些位置。任何时候，第一个元素（若存在）在位置0，下一个元素（若存在）在位置1，依此类推。字符串的长度是它的元素（即16位的值）的个数。空字符串长度为零，因而不包含任何元素。

不解释字符串内容的ECMAScript操作不会应用进一步的语义。对String值进行解释的操作将每个元素视为一个UTF-16代码单元。然而，ECMAScript并没有限制这些代码单元的值或它们之间的关系，所以进一步将String内容解释为UTF-16编码的Unicode代码点序列的操作必须考虑到不合法的子序列。这类操作对每一个数字值在0xD800到0xDBFF（Unicode标准定义为前导代理，或更正式地定义为高位代理代码单元）的代码单元和每一个数字值在0xDC00到0xDFFF（定义为尾随代理，或更正式地定义为低位代理代码单元）的代码单元采用以下规则进行特殊处理：

  • 既不是前导代理也不是尾随代理的代码单元，被解释为具有相同值的代码点。
  • 一个由两个代码单元组成的序列，如果第一个代码单元c1是前导代理，第二个代码单元c2是尾随代理，则它是一个代理对，并且会被解释为值为(c1 - 0xD800) × 0x400 + (c2 - 0xDC00) + 0x10000的代码点（见10.1.2）
  • 一个是前导代理或者是尾随代理的代码单元，如果不是代理对的一部分，则会被解释为具有相同值的代码点。

函数String.prototype.normalize（见21.1.3.13）可以用来明确地规范化一个字符串值。String.prototype.localeCompare（见21.1.3.10）在内部对String值进行规范化处理，但是没有其他操作隐含地对它们所操作的字符串进行规范化处理。只有那些被明确指定为对语言或地区敏感的操作才会产生语言敏感的结果。

注：这样设计的理由是为了使字符串的实现尽可能简单和高性能。如果ECMAScript的源文本是规范形式C，只要不包含任何Unicode转义序列，就能保证字符串字面值也是规范化的。

在本规范中，“A, B…的字符串连接”（其中每个参数是一个字符串值，或一个代码单元，又或是代码单元序列）表示一个字符串值，其代码单元序列是每个参数的代码单元（按顺序）的连接。

6.1.5 Symbol 类型

Symbol类型 是所有用来作为对象属性键的非字符串值的集合。(6.1.7)
每个可能的Symbol值是唯一的、不可变的。
每个Symbol值不变地持有一个关联值[[Description]]，该值是undefined或者一个字符串值。

6.1.6.1.3 Number::exponentiate ( base, exponent )

返回一个与实现有关的base的exponent次幂的结果的近似值。

  • 如果exponent为NaN，结果为NaN
  • 如果exponent为+0，结果为1，即使base为NaN
  • 如果exponent为-0，结果为1，即使base为NaN
  • 如果base为NaN并且exponent为非零，结果为NaN
  • 如果abs(base) > 1并且exponent为+∞，结果为+∞
  • 如果abs(base) > 1并且exponent为-∞，结果为+0
  • 如果abs(base)为1并且exponent为+∞，结果为NaN
  • 如果abs(base)为1并且exponent为-∞，结果为NaN
  • 如果abs(base) < 1并且exponent为+∞，结果为+0
  • 如果abs(base) < 1并且exponent为-∞，结果为+∞
  • 如果base为+∞并且exponent > 0，结果为+∞
  • 如果base为+∞并且exponent < 0，结果为+0
  • 如果base为-∞并且exponent > 0并且exponent是奇数，结果为-∞
  • 如果base为-∞并且exponent > 0并且exponent不是奇数，结果为+∞
  • 如果base为-∞并且exponent < 0并且exponent是奇数，结果为-0
  • 如果base为-∞并且exponent < 0并且exponent不是奇数，结果为+0
  • 如果base为+0并且exponent > 0，结果为+0
  • 如果base为+0并且exponent < 0，结果为+∞
  • 如果base为-0并且exponent > 0并且exponent是奇数，结果为-0
  • 如果base为-0并且exponent > 0并且exponent不是奇数，结果为+0
  • 如果base为-0并且exponent < 0并且exponent是奇数，结果为-∞
  • 如果base为-0并且exponent < 0并且exponent不是奇数，结果为+∞
  • 如果base < 0并且base为有限值并且exponent为有限值并且exponent不是整数，结果为NaN

注意：当base为1或-1，并且exponent为+Infinity或-Infinity时，base ** exponent的结果与IEEE 754-2008不同。ECMAScript的第一版为此操作指定了NaN结果，而IEEE 754-2008的更高版本则指定了1。出于兼容性原因，保留了历史ECMAScript行为。

6.1.6.1.4 Number::multiply ( x, y )

* MultiplicativeOperator执行乘法，产生其操作数的乘积。 乘法是可交换的。 由于有限的精度，乘法在ECMAScript中并不总是可结合的。

浮点乘法的结果受IEEE 754-2008二进制双精度算法的规则支配：

  1. 如果任一操作数为NaN，则结果为NaN。
  2. 如果两个操作数具有相同的符号，则结果的符号为正；如果两个操作数具有不同的符号，则结果为负。
  3. 无穷大乘以零会导致NaN。
  4. 无穷大与无穷大相乘会导致无穷大。 符号由上面已经陈述的规则确定。
  5. 无穷大与非零有限值的乘积导致有符号无穷大。 符号由上面已经陈述的规则确定。
  6. 在其余情况下，不涉及无穷大或NaN时，将使用IEEE 754-2008最近舍入模式计算乘积并将其舍入为最接近的可表示值。 如果模太大而无法表示，那么结果将是无穷大的适当符号。 如果模太小而无法表示，则结果为适当符号的零。 ECMAScript语言需要支持IEEE 754-2008定义的渐进式下溢。
6.1.6.1.5 Number::divide ( x, y )

/ MultiplicativeOperator执行除法，产生其操作数的商。 左边的操作数是被除数，右边的操作数是除数。 ECMAScript不执行整数除法。 所有除法运算的操作数和结果均为双精度浮点数。 除法结果由IEEE 754-2008算法的规范确定：

  1. 如果任一操作数为NaN，则结果为NaN。
  2. 如果两个操作数具有相同的符号，则结果的符号为正；如果两个操作数具有不同的符号，则结果为负。
  3. 无穷大除以无穷大会得到NaN。
  4. 无穷大除以零会导致无穷大。符号由上述规则确定。
  5. 无穷大除以非零有限值会导致有符号无穷大。符号由上述规则确定。
  6. 有限值除以无穷大将得出零。符号由上述规则确定。
  7. 将零除以零会产生NaN；将零除以任何其他有限值将得出零，其符号由上述规则确定。
  8. 非零有限值除以零会导致有符号无穷大。该标志由上述规则确定。
  9. 在其余情况下，既不涉及无穷大，也不涉及零，也不涉及NaN，则使用IEEE 754-2008舍入到最接近模式计算商并舍入到最接近的可表示值。如果模太大而无法表示，则操作会溢出；结果就是适当符号的无穷大。如果模太小而无法表示，则操作会下溢，并且结果为适当符号的零。 ECMAScript语言需要支持IEEE 754-2008定义的渐进式下溢。
6.1.6.1.6 Number::remainder ( n, d )

% MultiplicativeOperator从隐含的除法运算中得到余数； 左操作数是被除数，右操作数是除数。

注意：在C和C++中，取余运算符仅接受整数操作数。 在ECMAScript中，它也接受浮点操作数。

由％运算符计算的浮点余数运算的结果与IEEE 754-2008定义的“余数”运算不同。 IEEE 754-2008“余数”运算从舍入除法而不是截断除法计算余数，因此其行为与通常的整数余数运算符的行为不相似。 相反，ECMAScript语言在浮点运算上定义了％，其行为类似于Java整数余数运算符； 可以将其与C库函数fmod进行比较。

ECMAScript浮点余数运算的结果由IEEE算术规则确定：

  1. 如果任一操作数为NaN，则结果为NaN。
  2. 结果的符号等于被除数的符号。
  3. 如果被除数是无穷大，或者除数是零，或者两者都是，则结果为NaN。
  4. 如果被除数是有限的，并且除数是无穷大，则结果等于被除数。
  5. 如果被除数为零，并且除数为非零且有限，则结果与被除数相同。
  6. 在其余情况下，既不涉及无穷大，也不涉及零，也不涉及NaN，则被除数n和除数d的浮点余数r由数学关系r = n − (d × q)定义，其中q 是一个整数，仅当n / d为负时才为负，而只有当n / d为正时才为正，并且其大小应尽可能大，而不会超过n和d的真数学商。 使用IEEE 754-2008最近舍入模式计算r并舍入到最接近的可表示值。
6.1.6.1.7 Number::add ( x, y )

当将+运算符应用于两个数字类型的操作数时，它们将执行加法运算，从而产生操作数之和。 -运算符执行减法运算，产生两个数字操作数的差。

加法是可交换的运算，但并不总是可结合的。

加法的结果使用IEEE 754-2008二进制双精度算法的规则确定：

  • 如果任一操作数为NaN，则结果为NaN。
  • 两个相反符号的无穷大之和为NaN
  • 两个相同符号的无穷大之和为该符号的无穷大。
  • 无穷大与有限值之和等于该符号的无穷大
  • 两个负零之和为-0。两个正零或两个符号相反的零的和为+0
  • 零和非零有限值之和等于该非零操作数
  • 模相同符号相反的两个非零有限值的和为+0
  • 在其余情况下，既不涉及无穷大，也不涉及零，也不涉及NaN，并且操作数具有相同的符号或有不同的模，则使用IEEE 754-2008最近舍入计算出和并舍入到最接近的可表示值。如果模太大而无法表示，则操作会溢出，结果将是适当符号的无穷大。 ECMAScript语言需要支持IEEE 754-2008定义的渐进式下溢。

6.1.7 Object类型

Object 在逻辑上是一个属性的集合。每个属性既可以是一个数据属性，也可以是一个访问器属性：

  • 一个数据属性将一个键与一个ECMAScript语言值和一组布尔特性联系起来。
  • 一个访问器属性将一个键与一个或两个访问器函数，以及一组布尔特性联系起来。访问器函数用于存储或检索与该属性相关的ECMAScript语言值。

属性使用键来标识。一个属性键要么是一个ECMAScript字符串值，要么是一个Symbol值。所有String和Symbol值，包括空字符串，都可以用作属性键。属性名是指字符串属性键。

整数索引是一个字符串属性键，它是规范的数字字符串（请参见7.1.21），其数值为+0或≤2^53-1的正整数。 数组索引是整数索引，其数值i在+0 ≤ i <2^32-1范围内。

属性键被用来访问属性及其值。对属性的访问有两种：get和set，分别对应于值的检索和赋值。通过get和set的属性包括作为一个对象的直接组成部分的自有属性和继承属性，后者是由另一个关联对象通过属性继承关系提供。继承的属性可以是关联对象的自有属性或继承属性。一个对象的每个自有属性都必须有一个与该对象的其他自有属性的键值不同的键。

所有的对象在逻辑上都是属性的集合，但是有多种形式的对象在访问和操作其属性的语义上有所不同。请参阅6.1.7.2关于对象的多种形式的定义。

6.1.7.2 Object内部方法和内部槽

ECMAScript中对象的实际语义是通过称为内部方法的算法指定的。 ECMAScript引擎中的每个对象都与一组内部方法相关联，这些内部方法定义了其运行时行为。 这些内部方法不是ECMAScript语言的一部分。 它们仅出于说明目的由本规范定义。 但是，ECMAScript实现中的每个对象必须按照与其关联的内部方法指定的方式运行。 实现的确切方式由实现方式决定。

内部方法名称是多态的。 这意味着，当一个通用的内部方法名称被调用时，不同的对象值可能执行不同的算法。 调用内部方法的实际对象是调用的“目标”。 如果在运行时算法的实现尝试使用该对象不支持的对象的内部方法，则将引发TypeError异常。

内部槽对应于与对象关联并由各种ECMAScript规范算法使用的内部状态。 内部槽不是对象属性，也不会被继承。 根据特定的内部槽规范，这种状态可以由任何ECMAScript语言类型的值或特定的ECMAScript规范类型的值组成。 除非另有明确说明，否则内部槽将作为创建对象过程的一部分进行分配，并且可能不会动态添加到对象中。 除非另有说明，否则内部槽的初始值为未定义的值。 本规范中的各种算法都会创建具有内部槽的对象。 但是，ECMAScript语言没有提供将内部槽与对象关联的直接方法。

提供的任何实现都需要使用外来对象，这些对象也必须维护这些不变量。 违反这些不变量可能会导致ECMAScript代码具有无法预测的行为并造成安全问题。 但是，违反这些不变量绝不能损害实现的内存安全性。

一个实现必须不允许以任何方式规避这些不变量，例如通过提供实现基本内部方法功能而不使用其不变量的替代接口。

6.2 ECMAScript规范类型

规范类型 对应的是 元值，它被用于在算法中描述 ECMAScript 语言结构的语义和 ECMAScript语言类型。规范类型包括引用、列表、完结、属性描述式、词法环境、 环境纪录、数据块。规范类型的值是规范自造的，在 ECMAScript 的实现中不必对应到特定的实体上。规范类型可用来描述 ECMAScript 表达式运算的中间结果，但这样的值不能储存为对象的属性或 ECMAScript 语言的变量值。

6.2.1 列表和记录规范类型

列表类型用于说明 new 表达式、函数调用，和其他需要值的简单有序列表的算法中的参数列表的计算。列表类型的值是包含独立值的列表元素的简单有序序列。这些序列可以是任意长度。列表的元素可以使用0开始的索引进行随机访问。 为了符号上的方便，可以使用类似数组的语法来访问列表元素。 例如，arguments[2]是表示列表arguments的第3个元素的简写。

为方便起见，在本规范中，可以使用字面值语法来表示新的List值。 例如，«1、2»定义具有两个元素的List值，每个元素都被初始化为特定值。 一个新的空列表可以表示为«»。

记录类型用于描述本规范算法内的数据聚合。 Record类型值包含一个或多个命名字段。 每个字段的值可以是ECMAScript值，也可以是由与Record类型关联的名称表示的抽象值。 字段名称始终用双括号括起来，例如[[value]]。

为方便起见，在本规范中，可以使用类似于对象常量的语法来表示Record值。 例如，{[[field1]]: 42, [[field2]]: false, [[field3]]: empty}定义了一个Record值，该值具有三个字段，每个字段都被初始化为特定值。 字段名称顺序不重要。 没有明确列出的任何字段都被视为不存在。

在规范文本和算法中，点符号可用于指代Record值的特定字段。 例如，如果R是上一段中显示的记录，则R.[[field2]]是“ R的名为[[field2]]的字段”的简写。

常用Record字段的组合这种模式可能被命名，并且该名称可以用作字面Record值的前缀，以标识所描述的特定类型的聚合。 例如：PropertyDescriptor{[[Value]]: 42, [[Writable]]: false, [[Configurable]]: true}。

完结类型是一种记录，用于说明值的运行时传播和控制流中执行非局部控制转移的语句（break, continue, return, throw）的行为。

6.2.3.1 Await

算法步骤：

  1. 令completion为Await(value)

意味着：

  1. 令asyncContext为运行中的执行上下文
  2. 令promise为? PromiseResolve(%Promise%, value)
  3. 令stepsFulfilled为Await Fulfilled Functions中定义的算法步骤
  4. 令onFulfilled为! CreateBuiltinFunction(stepsFulfilled, « [[AsyncContext]] »)
  5. 将onFulfilled.[[AsyncContext]]设为asyncContext
  6. 令stepsRejected为Await Rejected Functions中定义的算法步骤
  7. 令onRejected为! CreateBuiltinFunction(stepsRejected, « [[AsyncContext]] »)
  8. 将onRejected.[[AsyncContext]]设为asyncContext
  9. 执行! PerformPromiseThen(promise, onFulfilled, onRejected)
  10. 从执行上下文栈中移除asyncContext，并将执行上下文栈的栈顶恢复为运行中的执行上下文
  11. 设置asyncContext的代码计算状态，使得当计算以完成记录completion恢复时，调用Await的算法的接下来的步骤将被执行，completion可用。
  12. 返回
  13. 注：这将返回到最早恢复计算asyncContext的操作的计算

其中上述步骤中的所有变量，除了completion之外，都是临时的，只在与Await有关的步骤中可见。

注：Await可以与前缀?或!结合使用，例如

  1. 令result为? Await(value)

意味着：

  1. 令result为Await(value)
  2. ReturnIfAbrupt(result)
Await Fulfilled Functions

Await fulfilled function是一个匿名的内置函数，它作为Await规范device的一部分，以常规完结的方式向调用者传递promise的兑现值。每个Await fulfilled function都有一个[[AsyncContext]]内部槽。

当使用参数value调用Await fulfilled function时，执行以下步骤：

  1. 令F为激活的函数对象
  2. 令asyncContext为F.[[AsyncContext]]
  3. 令prevContext为运行中的执行上下文
  4. 暂停prevContext
  5. 将asyncContext推入执行上下文栈，asyncContext现在是运行中的执行上下文
  6. 使用NormalCompletion(value)恢复暂停的asyncContext的计算，作为暂停该操作的结果。
  7. 断言：当我们到达此步骤时，asyncContext已经被从执行上下文栈中移除，prevContext现在是运行中的执行上下文
  8. 返回undefined

Await fulfilled function的"length"属性为1

6.2.3.1.2 Await Rejected Functions

Await rejected function是一个匿名的内置函数，它作为Await规范device的一部分，以非常规throw完结的方式向调用者传递promise的拒绝值。每个Await rejected function都有一个[[AsyncContext]]内部槽。

当使用参数reason调用Await rejected function时，执行以下步骤：

  1. 令F为激活的函数对象
  2. 令asyncContext为F.[[AsyncContext]]
  3. 令prevContext为运行中的执行上下文
  4. 暂停prevContext
  5. 将asyncContext推入执行上下文栈，asyncContext现在是运行中的执行上下文
  6. 使用ThrowCompletion(reason)恢复暂停的asyncContext的计算，作为暂停该操作的结果。
  7. 断言：当我们到达此步骤时，asyncContext已经被从执行上下文栈中移除，prevContext现在是运行中的执行上下文
  8. 返回undefined

Await rejected function的"length"属性为1

6.2.3.2 NormalCompletion

抽象操作NormalCompletion接受单个参数argument，例如：

  1. 返回 NormalCompletion(argument)

它是下面这种形式的简写：

  1. 返回Completion{[[Type]]: normal, [[Value]]: argument, [[Target]]:empty}

6.2.2.2 隐式完结值

本规范的算法通常隐式返回[[type]]为 normal 的完结记录。 除非从上下文中可以明显看出，否则返回一个而非完结记录的值的算法语句，例如

返回Infinity

实际上与以下内容含义相同：

返回NormalCompletion("Infinity")

但是，如果“ return”语句的值表达式是“完结记录”构造字面量，则返回作为结果的“完结记录”。 如果值表达式是对抽象操作的调用，则“ return”语句仅返回由抽象操作产生的完结记录。

抽象操作 Completion(completionRecord)用于强调将返回先前计算的完结记录。 Completion抽象操作接受单个参数completionRecord并执行以下步骤：

  1. Assert: completementRecord是一个完结记录。
  2. 将completementRecord作为此抽象操作的完结记录返回

在算法步骤中没有值的“ return”语句与以下内容具有相同的含义：

  1. 返回NormalCompletion(undefined)

任何在没有显式要求完整的完结记录值的上下文中对完结记录值的引用，等同于对完结记录值的[[value]]的字段的显式引用，除非完结记录是非常规完结。

6.2.2.3 抛出异常

抛出异常的算法步骤，例如

Throw a TypeError exception.

和以下内容含义相同：

返回Completion{[[type]]: throw, [[value]]: a newly created TypeError object, [[target]]:empty}.

6.2.2.4 ReturnIfAbrupt

算法步骤ReturnIfAbrupt(argument)和以下内容含义相同：

  1. Ifargument是非常规完结，返回argument
  2. Else ifargument是完结记录项，则令argument为argument.[[value]]

6.2.2.5 UpdateEmpty(completionRecord, value)

抽象操作UpdateEmpty接受参数completionRecord和value，然后执行以下步骤：

  1. Assert: 如果completionRecord.[[type]]为throw，则completionRecord.[[value]]不为empty
  2. IfcompletionRecord.[[type]]为throw，返回Completion(completionRecord)
  3. IfcompletionRecord.[[value]]不为empty，返回Completion(completionRecord)
  4. 返回Completion{[[type]]: completionRecord.[[type]], [[value]]: value, [[target]]: completionRecord.[[target]] }.

6.2.3 引用规范类型

注意：Reference类型用于解释诸如delete，typeof，赋值运算符，super关键字和其他语言功能之类的运算符的行为。 例如，赋值运算中的左操作数应产生引用。

一个引用是个已解析的名字或属性绑定。它由三部分组成：基值、引用名称，和一个严格引用标志（布尔值）。 基值是 undefined、一个Object、一个Boolean、一个String、一个Number、一个Symbol，或者一个环境记录项中。基值是 undefined 表示此引用不可以解析为一个绑定。引用名称是一个字符串或Symbol。

super引用是一种引用，用于表示使用super关键字表示的名称绑定。 super引用有额外的thisValue部分，其基值永远不会是环境记录项。

本规范中使用以下抽象操作来访问引用的部分：

  • GetBase(V)：返回引用值 V 的基值部分。
  • GetReferencedName(V)：返回引用值 V 的引用名称部分。
  • IsStrictReference(V)：返回引用值 V 的严格引用标志部分。
  • HasPrimitiveBase(V)：如果基值的类型是 Boolean、String、Symbol、Number，那么返回 true。
  • IsPropertyReference(V)：如果基值是个 Object 或 HasPrimitiveBase(V) 是 true，那么返回 true；否则返回 false。
  • IsUnresolvableReference(V)：如果基值是 undefined 那么返回 true，否则返回 false。
  • IsSuperReference(V)：如果引用有thisValue部分则返回 true。

本规范使用以下抽象操作来操作引用：

6.2.4 属性描述符规范类型

属性描述符类型是用来解释对象属性具体操作的特性集。属性描述符类型的值是记录项。每个字段的名称是一个特性名并且它的值是一个相应的特性值，这些特性在【属性的特性】一节有说明。此外，任何字段都可能存在或不存在。在本规范中用于标记属性描述符记录的字面值描述的模式名称是PropertyDescriptor。

根据是否存在或使用了某些字段，属性描述符的值可进一步划分为数据属性描述符和访问器属性描述符。一个数据属性描述符里包括叫做 [[Value]] 或 [[Writable]] 的字段。一个访问器属性描述符里包括叫做 [[Get]] 或 [[Set]] 的字段。任何属性描述都可能有名为 [[Enumerable]] 和 [[Configurable]] 的字段。一个属性描述符不能同时是数据属性描述符和访问器属性描述符；但是，它可能二者都不是。一个通用属性描述符是，既不是数据属性描述符也不是访问器属性描述符的属性描述符值。一个完全填充属性描述符是访问器属性描述符或数据属性描述符，并且拥有【属性的特性】一节 表2 或 表3 里定义的所有属性特性对应的字段。

在本规范中使用以下的抽象操作来操作属性描述符值：

6.2.4.1 IsAccessorDescriptor(Desc)

当抽象操作IsAccessorDescriptor以属性描述符Desc为参数被调用时，执行以下步骤：

  1. 如果Desc是undefined，返回false
  2. 如果Desc.[[Get]]和Desc.[[Set]]都不存在，返回false
  3. 返回true

6.2.4.2 IsDataDescriptor(Desc)

当抽象操作IsDataDescriptor以属性描述符Desc为参数被调用时，执行以下步骤：

  1. 如果Desc是undefined，返回false
  2. 如果Desc.[[Value]]和Desc.[[Writable]]都不存在，返回false
  3. 返回true

6.2.4.3 IsGenericDescriptor(Desc)

当抽象操作IsGenericDescriptor以属性描述符Desc为参数被调用时，执行以下步骤：

  1. 如果Desc是undefined，返回false
  2. 如果IsAccessorDescriptor(Desc)和IsDataDescriptor(Desc)均为false，返回true
  3. 返回false

6.2.4.4 FromPropertyDescriptor(Desc)

当抽象操作FromPropertyDescriptor以属性描述符Desc为参数被调用时，执行以下步骤：

  1. 如果Desc是undefined，返回undefined
  2. 令obj为ObjectCreate(%ObjectPrototype%)
  3. 断言：obj是没有自有属性的可扩展普通对象
  4. 如果Desc有[[Value]]字段，则执行CreateDataProperty(obj, "value", Desc.[[Value]])
  5. 如果Desc有[[Writable]]字段，则执行CreateDataProperty(obj, "writable", Desc.[[Writable]])
  6. 如果Desc有[[Get]]字段，则执行CreateDataProperty(obj, "get", Desc.[[Get]])
  7. 如果Desc有[[Set]]字段，则执行CreateDataProperty(obj, "set", Desc.[[Set]])
  8. 如果Desc有[[Enumerable]]字段，则执行CreateDataProperty(obj, "enumerable", Desc.[[Enumerable]])
  9. 如果Desc有[[Configurable]]字段，则执行CreateDataProperty(obj , "configurable", Desc.[[Configurable]])
  10. 断言：上面所有的CreateDataProperty操作均返回true
  11. 返回obj

6.2.5.5 ToPropertyDescriptor(Obj)

当抽象操作ToPropertyDescriptor以对象Obj为参数被调用时，执行以下步骤：

  1. 如果Type(Obj)不是Object，抛出 TypeError 异常
  2. 令desc为一个没有字段的新的属性描述符。
  3. 令hasEnumerable为? HasProperty(Obj, "enumerable")
  4. 如果hasEnumerable为true，则
    1. 令enumerable为! ToBoolean(? Get(Obj, "enumerable"))
    2. 将desc.[[Enumerable]]设为enumerable
  5. 令hasConfigurable为? HasProperty(Obj, "configurable")
  6. 如果hasConfigurable为true，则
    1. 令configurable为! ToBoolean(? Get(Obj, "configurable"))
    2. 将desc.[[Configurable]]设为configurable
  7. 令hasValue为? HasProperty(Obj, "value")
  8. 如果hasValue为true，则
    1. 令value为? Get(Obj, "value")
    2. 将desc.[[Value]]设为value
  9. 令hasWritable为? HasProperty(Obj, "writable")
  10. 如果hasWritable为true，则
    1. 令writable为! ToBoolean(? Get(Obj, "writable"))
    2. 将desc.[[Writable]]设为writable
  11. 令hasGet为? HasProperty(Obj, "get")
  12. 如果hasGet为true，则
    1. 令getter为? Get(Obj, "get")
    2. 如果IsCallable(getter)为false，且getter不是undefined，抛出TypeError异常
    3. 将desc.[[Get]]设为getter
  13. 令hasSet为? HasProperty(Obj, "set")
  14. 如果hasSet为true，则
    1. 令setter为? Get(Obj, "set")
    2. 如果IsCallable(setter)为false，且setter不是undefined，抛出TypeError异常
    3. 将desc.[[Set]]设为setter
  15. 如果desc.[[Get]]或desc.[[Set]]存在，则
    1. 如果desc.[[Value]]或desc.[[Writable]]存在，抛出TypeError异常
  16. 返回desc

6.2.4.6 CompletePropertyDescriptor(Desc)

当抽象操作CompletePropertyDescriptor以属性描述符Desc为参数被调用时，执行以下步骤：

  1. ReturnIfAbrupt(Desc)
  2. 断言：Desc是一个属性描述符
  3. 令like为Record{[[Value]]: undefined, [[Writable]]: false, [[Get]]: undefined, [[Set]]: undefined, [[Enumerable]]: false, [[Configurable]]: false}
  4. IfIsGenericDescriptor(Desc)或IsDataDescriptor(Desc)为true，则
    1. 如果Desc没有[[Value]]字段，将Desc.[[Value]]设为like.[[Value]]
    2. 如果Desc没有[[Writable]]字段，将Desc.[[Writable]]设为like.[[Writable]]
  5. Else
    1. 如果Desc没有[[Get]]字段，将Desc.[[Get]]设为like.[[Get]]
    2. 如果Desc没有[[Set]]字段，将Desc.[[Set]]设为like.[[Set]]
  6. 如果Desc没有[[Enumerable]]字段，将Desc.[[Enumerable]]设为like.[[Enumerable]]
  7. 如果Desc没有[[Configurable]]字段，将Desc.[[Configurable]]设为like.[[Configurable]]
  8. 返回desc
