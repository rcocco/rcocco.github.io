Github Actions
2021-03-26|计算机软件使用问题

概述

GitHub Actions可帮助您在软件开发生命周期内自动执行任务。GitHub Actions是事件驱动的，你可以在发生指定事件后运行一系列命令。例如，每当有人为存储库创建pull请求时，你都可以自动运行一个命令，用来执行软件测试脚本。

组件

工作流

工作流（Workflows）是你为你的存储库添加的自动化过程。 工作流由一个或多个作业组成，可以由事件安排或触发。 工作流可以用来在GitHub上构建，测试，打包，发布或部署项目。
事件（Events）：事件是触发工作流的特定活动。例如，当有人将提交推送到存储库，或者创建了一个issue或pull请求时，活动可能源自GitHub。 你还可以使用repository dispatch webhook在发生外部事件时触发工作流。 有关可用于触发工作流的事件的完整列表，请参阅触发工作流的事件。

作业

作业（Jobs）是在同一执行机器上执行的一组步骤。 默认情况下，具有多个作业的工作流程将并行运行这些作业。你还可以配置工作流以按顺序运行作业。 例如，一个工作流可以有两个顺序作业来构建和测试代码，其中测试作业取决于构建作业的状态。 如果构建作业失败，则测试作业将不会运行。

步骤

步骤（Steps）是可以在作业中运行命令的单个任务。 步骤可以是一个动作或一个shell命令。作业中的每个步骤都在同一个执行机器上执行，从而允许该作业中的动作彼此共享数据。

动作

动作（Actions）是独立的命令，用来组合成步骤，以便创建作业。动作是工作流中最小的可移植构建块。 您可以创建自己的动作，也可以使用GitHub社区创建的动作。 要在工作流中使用动作，您必须在一个步骤中包含它。

执行机器

执行机器（Runners）是已安装GitHub Actions执行应用程序的服务器。 您可以使用GitHub托管的执行机器，也可以使用你自己托管的执行机器。执行机器监听可用作业，一次运行一个作业，并将进度，日志和结果报告回GitHub。 GitHub托管的执行机器基于Ubuntu Linux，Microsoft Windows和macOS，工作流中的每个作业都在全新的虚拟环境中运行。 有关GitHub托管的执行机器的信息，请参阅“关于GitHub托管的执行机器”。 如果您需要其他操作系统或需要特定的硬件配置，你可以使用自己的执行机器。有关自托管的执行机器的信息，请参阅“托管您自己的执行机器”。

创建工作流

Github Actions使用YAML语法定义事件、作业和步骤，这些YAML文件存储在你的代码存储库中，位于.github/workflows目录。

在上面的例子中，
name是可选的，它是工作流的名字，会出现在Github仓库的Action标签页
on用来指定触发工作流的事件，可以设置为仅在某分支、path或tag上运行工作流，具体语法请参阅Github Actions的工作流语法
jobs将在learn-github-actions工作流文件中运行的所有作业分组在一起。
check-bats-version:定义jobs部分中的某个作业的名称。
runs-on: ubuntu-latest将作业配置为在Ubuntu Linux执行机器上运行。 这意味着该作业将在GitHub托管的新虚拟机上执行。 有关使用其他运行程序的语法示例，请参阅“ GitHub Actions的工作流语法”。
steps:将check-bats-version作业中运行的所有步骤组合在一起。 嵌套在此部分下的每个项目都是单独的动作或shell命令。
uses: actions/checkout@v2中use关键字告诉作业从v2中检索名为action/checkout@v2的社区动作。 此动作将检出您当前的存储库并将其下载到执行机器，从而使您可以对代码执行动作（例如测试工具）。每当工作流针对存储库的代码运行时，都必须使用checkout动作，或者使用存储库中定义的动作。
uses: actions/setup-node@v1动作在执行机器上安装node软件包，让你能够使用npm命令。
run: npm install -g bats中run关键字告诉作业在运行机器上执行一个命令。这个例子中，你使用npm安装了bats软件包。
run: bats -v最后你运行了bats命令输出软件版本。
