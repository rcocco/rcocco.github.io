5.1 DOM Ranges介绍

StaticRange和Range对象（范围）代表了一个结点树中的一系列内容。每个范围都有一个开始(start)和一个结束(end)，它们是边界点。边界点是由结点和偏移量组成的元组。换句话说，一个范围代表一个结点树内两个边界点之间的一段内容。

与StaticRange对象不同，Range对象会受到结点树变化的影响。因此，它们也被称为实时范围。这样的变化不会使它们失效，并将试图确保它仍然代表同一块内容。必要的是，当它所代表的部分内容被修改时，一个实时范围本身可能作为结点树的变化的一部分被修改。

注意：详见insert和remove算法，normalize()方法，以及replace data和split算法。

响应节点树的变化而更新实时范围可能很昂贵。对于每个结点树的变化，所有受影响的Range对象都需要被更新。即使应用程序对一些实时范围不感兴趣，当变化发生时，它仍然必须耗费维持它们是最新的成本。

StaticRange对象是一个轻量级的范围，当节点树发生变化时不会更新。因此，它不需要像实时范围那样承担维护成本。

5.2 边界点

边界点是一个由结点和偏移量（一个非负整数）组成的元组。

注意：一个正确的边界点的偏移量将在0和边界点的结点的length之间，闭区间。

一个边界点（结点A、偏移A），相对于另一个边界点（结点B、偏移B）的位置是在前面(before)、相同(equal)，还是在之后(after)，取决于以下步骤：

  1. 断言：nodeA和nodeB有相同的根
  2. 如果nodeA是nodeB，则当offsetA是offsetB时返回euqal，当offsetA小于offsetB时返回before，当offsetA大于offsetB时返回after
  3. 如果nodeA在nodeB的后面，则当(nodeB, offsetB)相对于(nodeA, offsetA)的位置为before，则返回after，若位置为after，则返回before
  4. 如果nodeA是nodeB的祖先：
    1. 令child为nodeB
    2. 只要child不是nodeA的孩子，就将child设为它的父结点
    3. 如果child的索引小于offsetA，则返回after
  5. 返回before

5.3 AbstractRange接口

实现了AbstractRange接口的对象通常称为范围(ranges)

一个范围有两个关联的边界点——开始(start)和结束(end)

为方便起见，一个范围的开始结点是指它的开始边界点的结点，开始偏移是指它的开始边界点的偏移。结束结点是指它的结束边界点的结点，结束偏移是指它的结束边界点的偏移。

一个范围是折叠的，当它的开始结点和结束结点相同，且开始偏移和结束偏移相同。

5.5. Interface Range

实现了Range接口的对象通常称为实时范围。

修改树的算法（特别是插入、删除、替换数据和分割算法）会修改与该树关联的实时范围。

实时范围的根是指它的开始结点的根。

一个结点node被包含在一个实时范围中，当node的根是范围的根，且(node, 0)在范围的开始结点之后(after)，且(node, node的长度)在范围的结束结点之前(before)

一个结点node被部分包含在一个实时范围中，当它是实时范围的开始结点的包含祖先，且不是结束结点。或者正好相反。

注意：一些事实可以帮助更好地理解这些定义。

  • 人们认为在实时范围内的内容包括所有被包含的结点，加上可能是开始结点和结束结点的一些内容，如果这些结点是Text、ProcessingInstruction或Comment节点。
  • 被包含在实时范围内的节点通常不是连续的，因为被包含的结点的父结点并不总是被包含的。
  • 然而，被包含结点的后代是被包含的，如果两个兄弟被包含，位于它们之间的任何兄弟也被包含。
  • 一个实时范围的开始结点和结束结点永远不会被包含。
  • 第一个被包含的结点（如果有的话）总是在开始结点之后，最后一个被包含的结点总是等于(equal)或早于(before)结束结点的最后一个后代。
  • 当且仅当开始结点和结束结点不同时，存在一个部分包含的结点。
  • commonAncestorContainer属性值既不包含也不部分包含。
  • 如果开始结点是结束结点的祖先，公共包含祖先将是开始结点。它的孩子中有且只有一个是部分包含的。其他孩子是被包含的，当且仅当它在这个部分包含孩子之前(precedes)。如果结束结点是开始结点的祖先，则情况相反。
  • 如果开始结点不是结束节点的包含祖先，反过来也不是，则公共包含祖先将与它们都不同。它的孩子中有且只有两个是部分包含的，其他孩子是被包含的，当且仅当它在这两个部分包含孩子之间。

——————
对于Web开发者：

——————

  2. 只要container不是结束结点的包含祖先，就令container为container的父结点
  3. 返回container

4.1 DOM介绍

在其原始意义上，"The DOM"是一个用于访问和操作文档（特别是HTML和XML文档）的API。在本规范中，术语 "文档"被用于任何基于标记的资源，从短的静态文档到长的文章或带有丰富多媒体的报告，以及完全成熟的互动应用程序。

每个这样的文档都被表示为一个结点树。树中的一些结点可以有子结点，而另一些结点则始终是叶子结点。

4.2 结点树

Document、DocumentType、DocumentFragment、Element、Text、ProcessingInstruction和Comment对象（简称为结点）参与到一棵树中，简称为结点树。

结点树的约束如下，表示为结点类型和其允许的子结点之间的关系。

  • Document：按树顺序
    1. 零个或多个ProcessingInstruction或Comment结点
    2. 一个可选的DocumentType结点
    3. 零个或多个ProcessingInstruction或Comment结点
    4. 一个可选的Element结点
    5. 零个或多个ProcessingInstruction或Comment结点
  • DocumentFragment
  • Element：
    1. 零个或多个Element,Text, ProcessingInstruction或Comment结点
  • DocumentType
  • Text
  • ProcessingInstruction
  • Comment
    1. 无

要确定一个结点node的长度，请打开节点：

  • DocumentType
    1. 零
  • Text
  • ProcessingInstruction
  • Comment
    1. 它的data的长度
  • 其他结点
    1. 它的孩子的数量

如果一个结点的长度为零，则视为空结点。

4.2.1 文档树

文档树是根为Document结点的结点树。

一个Document结点的文档元素是父结点为此Document结点的Element结点，如果存在的话。否则为null

注意：根据结点树的限制，只能有一个这样的元素。

如果一个Element结点的根是Document结点，则称这个Element结点在一个文档树中。

如果一个Element结点在一个文档树中，则称这个Element结点在文档中。注意：术语【在文档中】不应该再被使用，它表明使用它的标准还没有更新影子树。

4.2.2 影子树

影子树是根为影子根的结点树

影子根总是通过它的宿主连接到另一个结点树。因此，一棵影子树永远不是单独的。影子根的宿主的结点树有时被称为光树。

注意：一棵影子树对应的光树本身也可以是一棵影子树。

如果一个Element结点的包含影子根是一个Document结点，那么它就是连接(connected)的。

4.2.2.1 槽

影子树包含零个或多个作为槽(slot)的Element结点。

注意：槽只能通过HTML的slot元素创建

槽有关联的name（字符串），除非另有说明，否则它是空串。

注意：影子树中按照树顺序的第一个name为空字符串的槽，有时被称为 “默认槽”。

一个槽有一个关联的分配结点（一个slottables列表）。 除非另有说明，否则为空。

4.2.2.2 Slottables

Element和Text结点是slottables

注意：一个槽也可以是slottable

slottable有关联的name（字符串），除非另有说明，否则它是空串。

slottable有关联的分配槽（null或slot）。 除非另有说明，否则为空。 如果分配的槽非null，则slottable是已分配的。

2 事件

2.1 DOM事件介绍

在整个web平台中，事件被派发(发送)到对象来表示某个事情的发生，如网络活动或用户交互。这些对象实现了EventTarget接口，因此可以通过调用addEventListener()添加事件监听器来观察事件。

事件监听器可以通过调用removeEventListener()方法并传递相同的参数来删除。

另外，事件监听器可以通过传递AbortSignal到addEventListener()并在拥有该信号的控制器上调用abort()来删除。

事件也是对象，并且实现了Event接口（或一个派生接口）。在上面的例子中，ev是事件。ev被作为参数传递给事件监听器的回调（通常是一个JavaScript函数，如上所示）。事件监听器键入事件的type属性值（上例中的 "load"）。事件的target属性值返回事件被派发到的对象（上面的obj）。

尽管事件通常由用户代理派发，作为用户交互或完成某些任务的结果，但应用程序可以通过使用通常被称为合成事件的方式来自行派发事件。

除了信号，事件有时也被用来让应用程序控制一个操作的下一步发生什么。例如，作为表单提交的一部分，一个type属性值为 "submit"的事件被派发。如果这个事件的preventDefault()方法被调用，表单提交将被终止。希望通过应用程序派发的事件（合成事件）利用这一功能的应用程序可以利用dispatchEvent()方法的返回值。

当一个事件被派发到一个参与了树的对象（例如，一个元素）时，它也可以到达该对象的祖先上的事件监听器。实际上，首先会按照树顺序，调用该对象的所有包含祖先的捕获(captrue)为true的事件监听器。然后，如果事件的bubbles为true，则会按照逆序树顺序，调用该对象的所有包含祖先的捕获为false的事件监听器。

2.8 观察事件监听器

一般来说，开发者并不期望事件监听器的存在可以被观察到。事件监听器的影响是由其回调决定的。也就是说，一个开发者添加一个无操作的事件监听器不会期望它有任何副作用。

不幸的是，一些事件API被设计成有效地实现它们需要观察事件监听器。这可以使监听器的存在被观察到，即使是空的监听器也会对应用程序的行为产生巨大的性能影响。例如，触摸和滚轮事件，可以用来阻止异步滚动。在某些情况下，这个问题可以通过指定事件只有在至少有一个非passive监听器的情况下才可以取消而得到缓解。例如，非passive的TouchEvent监听器必须阻止滚动，但如果所有的监听器都是被动的，那么可以通过使TouchEvent不可取消来允许滚动开始（这样对preventDefault()的调用就被忽略了）。因此，调度事件的代码能够观察到非passive的监听器的不存在，并使用它来清除被调度的事件的cancelable属性。

理想情况下，任何新的事件API被定义为不需要这个属性（使用public-script-coord@w3.org 来讨论）。

2.11 行动vs发生

一个事件表示一个发生，而不是一个行动。换句话说，它代表来自一个算法的通知，可以用来影响该算法的未来进程（例如，通过调用preventDefault()）。事件不能被用作导致某些算法开始运行的行动或启动器。这不是它们的作用。

注意：这里特别指出，因为以前的DOM迭代有一个与事件相关的 "默认行动 "的概念，给了人们所有错误的想法。事件不代表或导致行动，它们只能被用来影响正在进行的行动。

1.1 树

一个树(tree)是一个有限分层树状结构。树顺序(tree order)是指对树进行先序遍历/深度优先遍历。

一个参与(participate)了树的对象具有父母(parent)，父母可能是null也可能是一个对象。该对象也有孩子(children)，它是对象的有序集合。如果一个对象A的父母为对象B，那么它是对象B的孩子。

一个对象的根(root)，在父母为null时是它自己，否则对象的根是它父母的根。树的根是任意参与了树，且父母为null的对象。

对象A称为对象B的后代(descendant)，是说对象A是对象B的孩子，或者A是某个对象C的孩子，而C又是B的后代。

包含后代(inclusive descendant)，是指一个对象或它的某个后代

对象A称为对象B的祖先(ancestor)，当且仅当对象B是对象A的后代

对象A称为对象B的兄弟(sibling)，当今且当B和A有相同的非null父母

包含兄弟(inclusive sibling)，是指一个对象或它的某个兄弟

对象A在对象B的前面(preceding)，是指A和B在同一个树中，且按照树顺序A出现在B之前

对象A在对象B的后面(following)，是指A和B在同一个树中，且按照树顺序A出现在B后面

对象的第一个孩子(first child)，是指对象的第一个孩子，如果没有孩子则为null

对象的最后一个孩子(last child)，是指对象的最后一个孩子，如果没有孩子则为null

对象的前一个兄弟(previou sibling)，是指它前面的第一个兄弟，如果没有前面的兄弟则为null

对象的后一个兄弟(next sibling)，是指它后面的第一个兄弟，如果没有后面的兄弟则为null

对象的索引(index)，是指它前面兄弟的数量，如果没有前面的兄弟则为0

为了简单起见，有时可能会在指代向用户呈现文档的方式时使用shown、displayed和visible等术语。这些术语并不意味着指代视觉媒介；必须考虑以同等方式将它们应用于其他媒体。

2.1.2 资源

本规范在提到用户代理是否具有能够解码外部资源语义的实现时，使用了支持(supported)这一术语。如果实现能够处理该格式或类型的外部资源，而不会忽略该资源的关键方面，则该格式或类型被认为是被支持的。一个特定的资源是否被支持，可能取决于该资源的格式的哪些功能被使用。

例如：如果一个PNG图像的像素数据可以被解码和渲染，那么它将被认为是被支持的格式，即使实现者不知道该图像还包含动画数据。

例如：如果所使用的压缩格式不被支持，那么MPEG-4视频文件就不会被认为是被支持的格式，即使实现可以从文件的元数据中确定电影的尺寸。

一些规范，特别是HTTP规范中，所说的表示(representation)，在本规范中称为资源(resource)。

一个资源的关键子资源(critical subresources)是指必须可用才能被正确处理的资源。哪些资源被认为是关键的或不关键的，由定义资源格式的规范来定义。

对于CSS样式表，我们暂且定义它们的关键子资源是通过@import规则导入的其他样式表，包括那些由其他导入的样式表间接导入的样式表。

这个定义并不完全具有互操作性；此外，一些用户代理似乎将背景图像或网页字体等资源算作关键子资源。理想情况下，CSS工作组会对此进行定义；参见w3c/csswg-drafts issue #1088以跟踪这方面的进展。

2.1.4 DOM树

当说明某些元素或特性被忽略(ignored)，或被作为其他值对待，或被当作其他东西处理时，这仅指结点处于DOM中之后的处理。在这种情况下，用户代理不得对DOM进行突变。

只有当一个内容特性的新值与它以前的值不同时，才称为内容特性改变了值；将一个特性设置为它已经拥有的值并不会改变它。

术语空值(empty)，当用于一个特性值、Text节点或字符串时，意味着文本的长度为零（即甚至不包含controls U+007F到U+009F，或U+0020 SPACE）。

结点A被插入到结点B中，是说使用参数A调用插入步骤，使得A的新父节点是B。同样，从结点B中移除结点A是说，以A为removementNode参数、B为oldParent参数调用移除步骤，B为oldParent参数调用移除步骤时，一个节点A被从一个节点B中移除。

一个结点被插入到文档中，是说使用该结点作为参数调用插入步骤，使得该结点现在处于文档树中。类似地，一个结点从文档中移除是说，以该结点为参数调用移除步骤，使得它现在不在文档树中。

一个结点变成连接的，是说以它为参数调用插入步骤，使得它现在是连接的。一个结点变成非连接的，是说以它为参数调用移除步骤，使得它现在不再是连接的。

一个结点是浏览上下文连接的，是说它是连接的，且它的shadow-including root的浏览上下文是非null。一个结点变成浏览上下文连接的，是说以它为参数调用插入步骤，使得它现在是浏览上下文连接的。一个结点变成浏览上下文非连接的，是说要么以它为参数调用移除步骤使得它不再是浏览上下文连接的，要么它的shadow-including root的浏览上下文变成null

2.1.5 脚本编写

构造器 “一个Foo对象”，这里Foo实际上是一个接口，有时用于代替更精确的说法 “一个实现Foo接口的对象”。

一个IDL特性被称为getting，当它的值被检索（例如被作者脚本）。当用新值为它赋值时称为setting

如果一个DOM对象被说成是live的，那么该对象上的特性和方法必须对实际的基础数据而不是数据的快照进行操作。

1.7 设计说明

本节为非规范性内容。

必须承认，HTML的许多方面乍一看似乎是荒谬和不一致的。

HTML及其支持的DOM API，以及许多支持的技术，都是在数十年间由一系列具有不同优先级的人开发的，在许多情况下，他们并不知道对方的存在。

因此，功能来自于许多方面，而且并不总是以特别一致的方式设计。此外，由于网络的特殊性，实现错误往往成为事实上的标准，而现在则是法律上的标准，因为内容往往是在无意中写成的，在它们被修复之前就依赖于它们。

尽管这样，人们还是努力坚持某些设计目标。这些将在接下来的几个小节中描述。

1.7.3 可扩展性

本节为非规范性内容。

HTML具有广泛的可扩展性机制，可用于以安全的方式添加语义。

  • 作者可以使用class属性来扩展元素，有效地创建他们自己的元素，同时使用最适用的现有 "真正的 "HTML元素，这样浏览器和其他不知道该扩展的工具仍然可以在一定程度上支持它。例如，这就是微格式(microformats)所使用的方法。
  • 作者可以使用data-*=""属性为内联的客户端脚本或服务器端的网站脚本提供数据处理。这些属性保证不会被浏览器触及，并允许脚本在HTML元素上包含数据，然后脚本可以查找和处理这些数据。
  • 作者可以使用<meta name="" content="">机制来包含整个页面的元数据。
  • 作者可以使用rel=""机制，通过注册扩展到预定义的链接类型集，来为链接注释特定的含义。这也被微格式所使用。
  • 作者可以使用<script type="">机制嵌入原始数据，并使用自定义类型，以便由内联或服务器端的脚本进一步处理。
  • 作者可以创建插件并使用embed元素来调用它们。这就是Flash的工作原理。
  • 作者可以使用JavaScript原型设计机制来扩展API。例如，这被脚本库所广泛使用。
  • 作者可以使用微数据功能 （itemscope=""和itemprop=""属性）来嵌入嵌套的数据名-值对，以便与其他应用程序和网站共享。

1.8 HTML vs XML 语法

本节为非规范性内容。

HTML规范定义了一种用于描述文档和应用的抽象语言，以及一些用于与使用这种语言的资源的内存表示进行交互的应用编程接口(API)。

内存中的表示被称为DOM HTML，简称DOM

有各种具体的语法可以用来传输使用这种抽象语言的资源，本规范中定义了其中的两种。

第一个这样的具体语法是HTML语法。这是大多数作者建议使用的格式。它与大多数旧版 Web 浏览器兼容。如果一个文档以text/html MIME类型传输，那么它将被Web浏览器当作HTML文档来处理。本规范定义了最新的HTML语法，简称为 “HTML”。

第二种具体的语法是XML。当一个文档以XML MIME类型（如application/xhtml+xml）传输时，那么它就会被Web浏览器当作XML文档来处理，由XML处理器进行解析。请作者注意，对XML和HTML的处理是不同的；特别是，即使是轻微的语法错误，也会阻止标记为XML的文档被完全呈现，而在HTML语法中，这些错误会被忽略。

注：HTML的XML语法以前被称为 “XHTML”，但本规范没有使用这个术语（除其他原因外，还因为MathML和SVG的HTML语法没有使用这个术语）。

DOM、HTML语法和XML语法不能都表示相同的内容。例如，命名空间不能用HTML语法表示，但在DOM和XML语法中都支持。同样，使用noscript功能的文档可以用HTML语法表示，但不能用DOM或XML语法表示。包含字符串"-->"的注释只能用DOM表示，不能用HTML和XML语法表示。
